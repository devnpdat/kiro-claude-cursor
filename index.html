5W1H về
Claude Code, Kiro & Cursor

Hiểu đúng 11 khái niệm cốt lõi của 3 AI IDE — và cách tổ chức workspace để cả 3 cùng đọc 1 nguồn knowledge, không duplicate.

CLAUDE.md AGENTS.md Steering Command Skill Hook MCP Subagent Spec Power Cursor Rules

❓ Vì sao bạn đang đọc trang này?

  • ❌ Steering bạn đang dùng — có thể đang sai mà không biết.
  • ❌ Dùng /review-fix chạy 10 lần ra 10 lỗi khác nhau.
  • ❌ AI mỗi ngày đẻ ra thêm 1 khái niệm mới — không theo kịp.
  • ❌ Team có người dùng Claude, người dùng Kiro, người dùng Cursor — phải copy rule cho 3 nơi?

3 IDE — 1 Workspace

Mỗi tool có thế mạnh riêng. Tổ chức đúng → cả team thoải mái chọn.

🤖 Claude Code

CLI + IDE plugin (Rider, VS Code, JetBrains)
  • Anthropic native — model mạnh nhất Claude
  • Tool use + MCP cực mạnh
  • Subagent, Skill, Hook ecosystem
  • Slash command tự custom
  • Strong for: agentic work, refactor lớn

🚀 Kiro

IDE đầy đủ — fork VS Code + AI layer
  • Spec workflow (req → design → tasks)
  • Steering modes (always/fileMatch/manual)
  • Hooks: button-triggered + event-based
  • Power = plugin 1-click (Figma, Postman)
  • Strong for: feature lớn cần kiểm soát

🖱️ Cursor

IDE fork VS Code — AI tab completion vô địch
  • Tab completion thông minh nhất hiện tại
  • Composer = agent mode (multi-file edit)
  • @ context: @Files, @Code, @Docs, @Web
  • Rules .mdc modular theo file glob
  • Strong for: iterate UI/feature nhanh

1. CLAUDE.md

WHAT Cái gì?
File markdown ở ROOT project. Là "system prompt" tự load của Claude Code.
WHY Tại sao?
Cho AI biết ta là ai, code gì, convention gì. Tránh khởi tạo session "mù".
WHO Ai dùng?
Dev dùng Claude Code (CLI/IDE). Tạo riêng cho project của mình.
WHEN Khi nào?
MỖI session mới. Tự động load.
WHERE Ở đâu?
Tại ROOT project <project>/CLAUDE.md
HOW Như thế nào?
Viết ngắn gọn: identity, stack, conventions, reference qua @file.

✅ NÊN — Ngắn, focus hành vi

# Return Home BE
## Identity
- Service: API trả hàng (YOUR-CHAIN)
- Stack: .NET 8, EF Core, MediatR

## Conventions
- Entity: OrderReturn
- Async only — NO .Result
- [Authorize] mandatory

## References
@../docs/conventions.md

❌ KHÔNG NÊN — Nhồi nhét

  • Paste 30 trang API docs → Context bloat
  • Paste current sprint state, task progress → AI nói dối vì state cũ
  • Copy rule trùng với linter → Tốn focus
  • Viết mơ hồ "write good code" → Vô dụng

Rule of thumb: < 5KB.

2. AGENTS.md ✨ Universal — cho cả 3 IDE

WHAT Cái gì?
Chuẩn UNIVERSAL. File markdown mọi tool (Claude, Cursor, Aider, Kiro...) đều đọc.
WHY Tại sao?
Tránh duplicate context. Viết 1 lần, mọi tool hiểu.
WHO Ai dùng?
Team đa-tool, hoặc người muốn ổn định trước thay đổi AI.
WHEN Khi nào?
Khi team dùng > 1 AI tool song song.
WHERE Ở đâu?
ROOT project (cùng cấp CLAUDE.md)
HOW Như thế nào?
Tạo AGENTS.md chứa rules chung. CLAUDE.md gọi @AGENTS.md, .cursor/rules/main.mdc cũng reference nó.

3. Steering (Dành cho Kiro)

WHAT Cái gì?
Tài liệu định hướng HÀNH VI lâu dài trong Kiro. (Không phải database API).
WHY Tại sao?
Tách biệt "hành vi cố định" và "context tạm thời" qua mọi session.
WHEN Khi nào?
3 mode: Always (mọi lúc) • File Match (khớp pattern) • Manual (gõ #tag).
WHERE Ở đâu?
.kiro/steering/ (product, tech, structure.md)
HOW Như thế nào?
Frontmatter inclusion: always/fileMatch/manual. Giữ NGẮN. Tham chiếu qua #[[file]].

💡 Phân loại đúng Documents:

Steering (.kiro/steering)
Identity, Tech stack chính, Conventions ko linter, Quyết định cố định.
Docs (docs/)
API reference, Business rules detailed, ADRs.
Specs (specs/)
Requirements, Design, State/Progress (thay đổi liên tục).

4. Slash Command (/command-name)

WHAT Cái gì?
Workflow rút gọn. Gõ /name để chạy script prompt định sẵn.
WHY Tại sao?
Đóng gói workflow lặp lại, tránh paste prompt dài mỗi lần.
WHERE Ở đâu?
.claude/commands/<name>.md
HOW Như thế nào?
Tạo file markdown với frontmatter. Dùng biến $ARGUMENTS.

📄 Ví dụ: /spec-from-jira

---
description: Tạo spec từ Jira ticket
argument-hint: "<JIRA-ID>"
---
Tạo spec đầy đủ cho $ARGUMENTS.
Bước 1: Dùng MCP đọc Jira issue_key: $ARGUMENTS
Bước 2: Đọc Confluence linked docs
Bước 3: Tạo specs/{slug}/ (requirements, design, tasks)
Bước 4: Confirm trước khi save.

Cách dùng trong chat: /spec-from-jira FC-53161

5. Skill

WHAT Cái gì?
Module "kỹ năng" đóng gói. AI tự kích hoạt khi tình huống match.
WHY Tại sao?
User không cần nhớ lệnh (khác Command). Tốt cho capability đặc thù (pptx, excel).
WHERE Ở đâu?
.claude/skills/<name>/SKILL.md
HOW Như thế nào?
Folder chứa SKILL.md (mô tả trigger) + các script/template helper đi kèm.

📁 Cấu trúc một Skill (vd: tạo pptx)

pptx/ ├── SKILL.md ← Triggers + workflow ├── editing.md ← Sub-guide ├── scripts/ │ ├── thumbnail.py ← Helper scripts └── resources/ └── (templates...)

🎯 Skill vs Command

Command: User CHỦ ĐỘNG gõ. 1 prompt cố định. Cho workflow user-triggered.

Skill: AI TỰ kích hoạt khi cần. Gồm nhiều files/scripts. Tốt cho capability theo file type/domain (cần in sẵn best practice).

6. Hook

WHAT Cái gì?
Trigger TỰ ĐỘNG chạy action khi có event xảy ra. Hoặc nút bấm thủ công.
WHY Tại sao?
Tự động hóa boring tasks (format, lint). Đóng gói quy trình thành 1 nút.
WHERE Ở đâu?
.claude/settings.json hoặc .kiro/hooks/
HOW Như thế nào?
Định nghĩa matcher (event/regex) + action (shell command / askAgent).

⚠️ Đừng nhầm 3 loại Hook này

1

Claude Code Hook

Trong session Claude Code

.claude/settings.json

Use case: Auto-format (dotnet format) sau khi sửa file (PostToolUse).

2

Kiro Hook

User trigger thủ công / Event

.kiro/hooks/*.kiro.hook

Use case: Bấm nút để update docs, generate MR summary.

3

OS Cron Task

Chạy ngầm mức OS

crontab / launchd

Use case: Fetch data định kỳ bằng script bash (every Mon 9am).

7. MCP ✨ Universal — cho cả 3 IDE

WHAT Cái gì?
Giao thức chuẩn để AI giao tiếp với "tools bên ngoài" (Database, APIs).
WHY Tại sao?
Giúp AI truy cập Jira, Confluence, Git... mà không cần dev tự code. CÙNG 1 MCP server, cả 3 IDE đều xài được.
WHERE Ở đâu?
Claude: .claude/settings.json · Kiro: .kiro/settings/mcp.json · Cursor: .cursor/mcp.json
HOW Như thế nào?
Khai báo server URL & tokens. AI sẽ TỰ DISCOVER list tools.

⚙️ Cấu hình MCP (Ví dụ chung cho cả 3 IDE)

{
  "mcpServers": {
    "company-context": {
      "url": "https://mcp.domain.com/mcp",
      "headers": {
        "X-GitLab-Token": "${GITLAB_TOKEN}",
        "X-Jira-Token": "${JIRA_TOKEN}"
      }
    }
  }
}

⚠️ Lưu ý: Add mcp.json vào .gitignore, KHÔNG commit token lên Git!

🔄 Keycloak JWT hết hạn mỗi ngày — Giải pháp Auto-Refresh

Token trong .kiro/settings/mcp.json expire sau ~24h. Hệ thống auto-refresh gồm 3 trigger:

⏰ Scheduled — 7:30 sáng tự động

launchd gọi core script mỗi sáng trước giờ làm. Token luôn fresh khi bắt đầu ngày.

⌨️ On-demand — gõ 1 lệnh

Alias frt-refresh gọi core script bất kỳ lúc nào token hết hạn giữa chừng.

🔍 On-401 — auto detect không cần làm gì

Kiro hook + Claude Code PostToolUse detect lỗi 401/Unauthorized từ MCP call → tự gọi script → thông báo reconnect. Zero friction.

Cơ chế:

  1. Core script đọc refresh_token từ macOS Keychain (an toàn, không plain text)
  2. POST lên Keycloak endpoint → nhận access_token mới
  3. Update Authorization header trong .kiro/settings/mcp.json
  4. Chạy sync-mcp-kiro-to-claude.sh → sync sang ~/.claude.json

Setup (1 lần duy nhất):

# Nhập password 1 lần → lưu refresh_token vào Keychain → cài launchd + alias
bash ~/your-project/install-refresh-token-auto.sh

# Sau đó chỉ cần gõ khi cần on-demand
frt-refresh

Log: cat /tmp/frt-token-refresh.log · Script: ~/.claude/scripts/frt-refresh-token.sh

8. Subagent

WHAT Cái gì?
Agent CON với context riêng, tools giới hạn, system prompt chuyên biệt.
WHY Tại sao?
Tránh "làm bẩn" context chính khi cần deep dive (vd: audit security).
WHERE Ở đâu?
.claude/agents/<name>.md
HOW Như thế nào?
Gọi qua @subagent-name hoặc Claude tự delegate.

🎯 Subagent vs Skill

Skill = Capability (Cách làm một việc cụ thể).

Subagent = Persona (Góc nhìn chuyên môn - vd: Security Reviewer. Không sửa code, chỉ đọc/tư vấn).

9. Spec

WHAT Cái gì?
Quy trình có cấu trúc để xây tính năng phức tạp: req → design → tasks.
WHY Tại sao?
Ép AI hiểu YÊU CẦU trước khi code. Có checkpoint confirm.
WHO Ai dùng?
Dev / BA / TechLead xử lý feature vượt khả năng 1 prompt.
WHEN Khi nào?
Feature sửa > 5 file hoặc có business logic phức tạp.
WHERE Ở đâu?
.kiro/specs/<feature>/ (chứa req, design, tasks.md)
HOW Như thế nào?
Bắt đầu spec workflow → AI hỏi req → viết design → chia tasks.

10. Power (Kiro)

WHAT Cái gì?
Plugin "siêu năng lực" cài 1 click vào Kiro kết nối dịch vụ ngoài.
WHY Tại sao?
Setup cực nhanh, có UI, không cần cấu hình token tay phức tạp.
WHO Ai dùng?
Team non-tech (BA/Design) hoặc Dev muốn dùng tool có sẵn (Figma, Postman).
HOW Như thế nào?
Mở Command Palette → "Configure Powers" → Chọn plugin → Cài.

⚖️ Power vs MCP — Khác gì?

Tiêu chí Power MCP
Cài đặt 1 click qua UI Sửa JSON config tay
Setup Đã đóng gói sẵn Khai báo URL + token thủ công
UX Có giao diện Raw — dev-facing
Tính linh hoạt Chỉ các plugin có sẵn Bất kỳ MCP server nào (Jira, GitLab cty)

💡 Rule of thumb: Có Power → dùng Power (nhanh, đẹp). Không có Power → config MCP (linh hoạt, hệ thống nội bộ).

11. Cursor Rules (Cursor IDE — .cursor/rules/*.mdc)

WHAT Cái gì?
Tài liệu rules dạng modular cho Cursor IDE — mỗi file .mdc có frontmatter quyết định khi nào AI dùng.
WHY Tại sao?
Tab completion & Composer agent của Cursor cần "biết style" code mình. Rules = behavior layer của Cursor.
WHO Ai dùng?
Dev dùng Cursor IDE (fork VS Code). Đặc biệt khi cùng team dùng AI khác nhau.
WHEN Khi nào?
4 mode: alwaysApply (luôn) · globs (theo file pattern) · description (AI quyết) · Manual (gõ @rule-name).
WHERE Ở đâu?
.cursor/rules/*.mdc (mới — multiple files) hoặc .cursorrules (legacy — 1 file)
HOW Như thế nào?
Tạo .mdc files với frontmatter description, globs, alwaysApply. Tách rule theo domain (dotnet.mdc, frontend.mdc, security.mdc).

📄 Ví dụ: .cursor/rules/dotnet.mdc

---
description: Convention .NET BE return-home
globs:
  - "**/*.cs"
alwaysApply: false
---

# Convention .NET

- Entity name: `OrderReturn`
  (KHÔNG dùng OrderReturnHead)
- Async only — NO `.Result` / `.Wait()`
- `[Authorize]` mandatory mọi endpoint
- DTO không depend trực tiếp Entity domain

@docs/platform/conventions.md
@docs/api/return-home/

🔍 4 Mode kích hoạt — chọn đúng

alwaysApply: trueIdentity, branching convention
globs: ["**/*.cs"]Rule chỉ áp dụng cho file matching
description: "..."AI tự load khi context phù hợp
ManualUser gõ @rule-name vào prompt

💡 Mỗi rule = 1 concern. Đừng nhồi vào 1 file.

🎯 Cursor Rules vs Steering vs CLAUDE.md — cùng là "behavior layer", khác chỗ nào?

CLAUDE.md
1 file ROOT. Auto load mọi session. Đơn giản.
Steering (Kiro)
Nhiều file, 3 mode (always/fileMatch/manual). Frontmatter inclusion.
Cursor Rules
Nhiều .mdc file, 4 mode. Frontmatter alwaysApply/globs/description.

1

Gather context

Gọi execute_flow(b5_build) (MANDATORY — không fallback) → đọc Confluence linked → load steering rules

💡 Lợi ích

AI hiểu đúng business context từ Jira + design doc — không code theo hiểu nhầm. Bỏ bước này = risk implement sai nghiệp vụ.

2

Identify deps

💡 Lợi ích

3

Apply Karpathy + Sequential Thinking

sequentialthinking phân tích multi-step → surface assumptions A1, A2... → xác định surgical scope → define success criteria

→ Tạo specs/TICKET-XXX/plan.md: bảng APIs sorted Easy→Hard + section "Dev Review Notes"

💡 Lợi ích

🛑 STOP — Dev reviews plan

💡 Lợi ích

Dev là người chịu trách nhiệm cuối — checkpoint này đảm bảo AI không tự ý implement sai scope. Edit trực tiếp file thay vì chat ping-pong = nhanh hơn, rõ hơn.

5

Implement loop

1 atomic commit/API theo thứ tự plan.md → verify success criteria → update Status TODO→IN_PROGRESS→DONE

💡 Lợi ích

Atomic commit = dễ revert từng API nếu cần. Thứ tự Easy→Hard = unblock nhanh, tránh block cả sprint vì Hard API chưa xong. Status tracking = dev và AI luôn biết còn bao nhiêu việc.

6

MANDATORY Self-Review

Auto invoke nt-review skill → output docs/reviews/<branch>_review_<time>.md

FAIL ❌ → auto nt-fix-review → re-review → loop tới PASS ✅

AI KHÔNG báo "xong" khi verdict = FAIL

💡 Lợi ích

Catch lỗi [Required] nullable, DTO patterns V1-V6, out-of-scope changes trước khi sếp review. Tiết kiệm 1-2 vòng MR comment. Dev không cần nhớ checklist — AI tự lo.

🆕 Sequential Thinking (Phase 3):

AI suy nghĩ có cấu trúc, có thể revise lại assumption trước khi commit plan — output plan chất lượng hơn so với 1-shot thinking.

📋 2. MR Summary — Format Chuẩn (6 dòng)

✅ Format ĐÚNG (terse, plain text)

- Link merge: https://git.example.com/your-team/your-service/-/merge_requests/45
- Component: oms-service
- Từ nhánh : features/yourname-TICKET-123-add-required-fields
- Tới nhánh : ci
- Nội dung : Thêm [Required] vào Return Kafka ETO fields cho IMS mapping
- Link Task : https://your-jira.com/browse/TICKET-123

→ Paste thẳng vào task tool nội bộ. Sếp review nhanh.

❌ Format SAI (markdown, dài dòng)

  • Có # MR:, ## Links, ## Description
  • Có Test Checklist / Quality Checklist
  • Có Files Changed table
  • Có Stats (lines +/-)
  • "Nội dung" bằng English: "Implement validation..."
  • "Nội dung" không có động từ đầu: "Tính năng mới chặn..."

→ Sếp ngán đọc, không review nổi.

🎯 Rules quan trọng

Component:

Auto-detect từ git remote get-url origin. KHÔNG hardcode "oms-service" vô prompt.

Tới nhánh:

Default là ci. Khi pass CI → đổi sang release/yyyymmdd-vx.x.x-lc.

Nội dung:

Tiếng Việt, 5-15 từ, bắt đầu bằng động từ: Chặn / Thêm / Sửa / Fix / Refactor / Cập nhật / Tối ưu...

Link Task:

Multi-Jira OK. Detect từ branch name + commit messages. Comma-separated URLs.

🎓 4. Lessons Learned — Anti-Patterns Đã Gặp

❌ Anti-Pattern 1: Token leak

Tình huống: Paste mcp.json (kèm token) vào AI chat để hỏi config.

Hậu quả: Token vào API log Anthropic + có thể leak qua re-share.

✓ Đúng: Mask tokens trước khi paste. Nếu lỡ paste → revoke + rotate ngay (4 tokens: JWT, GitLab PAT, Confluence, Jira).

❌ Anti-Pattern 2: Fallback mandatory tool

Tình huống: Skill viết "execute_flow OR jira_get_issue if unavailable".

Hậu quả: AI thấy execute_flow lỗi → tự fallback → violate company rule.

✓ Đúng: Mandatory = MANDATORY. Lỗi → STOP, báo user, đợi explicit override.

❌ Anti-Pattern 3: Hardcode repo name

Tình huống: Hook ghi "repo oms-service (repos/oms-service)" trong prompt.

Hậu quả: Chạy ở return-home → AI vẫn nghĩ là oms-service → confused.

✓ Đúng: Detect dynamic qua git remote get-url origin.

❌ Anti-Pattern 4: Format quá phức tạp

Tình huống: MR Summary gen ra markdown 40+ dòng có checklist, table, stats.

Hậu quả: Không paste được vào task tool nội bộ (chỉ chấp nhận 6 dòng terse).

✓ Đúng: Check format SẾP/TEAM thực tế đang dùng trước khi gen. Đôi khi terse hơn = tốt hơn.

❌ Anti-Pattern 5: Skill chỉ cài 1 IDE

Tình huống: Cài skill .claude/skills/ → tưởng work cả 3 IDE.

Hậu quả: Khi switch sang Kiro/Cursor → skill invisible.

✓ Đúng: Tạo wrapper trong .kiro/steering/ + .cursor/rules/ trỏ về canonical.

❌ Anti-Pattern 6: MCP scope sai

Tình huống: claude mcp add trong 1 repo → tưởng global.

Hậu quả: Sang repo khác → MCP missing → AI không gọi Jira/Confluence được.

✓ Đúng: Dùng --scope user hoặc edit ~/.claude.json top-level mcpServers.

❌ Anti-Pattern 7: Reference: text vs @file mention

Tình huống: Command/skill có dòng > Reference: .kiro/steering/principles.md tưởng AI auto-load.

Hậu quả: AI không thực sự đọc file đó — chỉ là DOC pointer cho human. Output review thiếu rule, sếp review catch lỗi mới.

✓ Đúng: Dùng cú pháp @.kiro/steering/principles.md ở đầu file để Claude Code thực sự inject content vào context.

❌ Anti-Pattern 8: Slash command không cross-IDE

Tình huống: Skill feature-startup-protocol viết "MANDATORY chạy /nt-review". Anh chạy ở Kiro/Cursor.

Hậu quả: Kiro & Cursor KHÔNG có syntax /command — chỉ Claude Code có. AI ngó lơ Phase 6.

✓ Đúng: Tạo SKILL canonical (.claude/skills/nt-review/SKILL.md) + wrappers Kiro/Cursor. Slash command chỉ là quick-invoke wrapper. Reference từ skill khác dùng @file (universal) thay vì /command (Claude-only).

❌ Anti-Pattern 9: Override built-in command

Tình huống: Tạo user-defined /code-review trùng tên built-in của Claude Code.

Hậu quả: Confusion — list /skills hiện 2 entries cùng tên. User không biết cái nào sẽ chạy. Có thể override mất tính năng built-in (configurable effort, inline PR comment).

✓ Đúng: Kiểm tra Claude Code built-in trước khi đặt tên (gõ / xem autocomplete). Custom command nên có prefix riêng (vd: /nt-*) tránh đè.

❌ Anti-Pattern 10: Principles generic, không đủ specific

Tình huống: File techlead-review-principles.md chỉ có rule chung (CRITICAL/MAJOR/MINOR generic). Sếp catch lỗi [Required] DateTime? mà file principles không cover.

Hậu quả: Self-review pass nhưng sếp vẫn FAIL — vì AI không có "knowledge base" specific để catch các pattern stack/cty.

✓ Đúng: Mỗi lỗi sếp catch lần đầu → cập nhật principles ngay (vd: section DTO/Validation Patterns V1-V6). Lâu dài principles trở thành sentinel chống sếp catch lại.

3. Hook — Trigger tự động khi sự kiện xảy ra

Hook chạy khi có event (save file, push git, tool call lỗi...). Claude Code và Kiro có cú pháp khác nhau hoàn toàn.

🟣 Claude Code — settings.json

File: .claude/settings.json → hooks section. Ngôn ngữ: JSON + bash string.

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",     // regex match tool name
      "hooks": [{
        "type": "command",
        "command": "if [[ \"${file}\" == *.cs ]]; then
          dotnet format --include \"${file}\" 2>/dev/null || true
        fi"
      }]
    }],
    "UserPromptSubmit": [{
      "matcher": ".*",
      "hooks": [{
        "type": "command",
        "command": "if grep -qE 'glpat-|eyJhbG' <<< \"${prompt}\";
          then echo 'WARN: secret detected'; fi"
      }]
    }]
  }
}

Event types: PostToolUse · PreToolUse · UserPromptSubmit · Stop
Env vars: ${file} · ${prompt} · ${output} · ${tool_name}

🟡 Kiro — .kiro/hooks/<name>.kiro.hook

File: JSON. Kiro hook KHÔNG chạy bash trực tiếp — dùng askAgent để bảo AI làm gì.

{
  "name": "Auto Refresh Token on Expire",
  "version": "2",
  "description": "Detect 401 từ MCP call → refresh token",
  "when": {
    "type": "postToolUse",
    "toolTypes": [".*ai_context_engine.*"]  // regex
  },
  "then": {
    "type": "askAgent",
    "prompt": "Check if tool output contains '401' or
      'Unauthorized'. If YES: run
      `bash ~/.claude/scripts/frt-refresh-token.sh`,
      then tell user to reconnect MCP server.
      If NO: do nothing."
  }
}

when.type: userTriggered (nút bấm UI) · postToolUse (sau tool call)
then.type: chỉ có askAgent — Kiro không expose bash runner

4. Cross-IDE Compatibility — Bảng tra nhanh

Khi muốn 1 tính năng chạy trên cả 3 IDE — đây là cú pháp từng loại.

Muốn làm gì Claude Code Kiro Cursor

Slash command /command-name ✅ ❌ Không hỗ trợ → dùng steering ❌ Không hỗ trợ → dùng rule

✅ 5 Golden Rules để viết cross-IDE đúng

  1. Logic duy nhất ở SKILL.md — Kiro steering và Cursor .mdc chỉ là thin wrapper trỏ về canonical. Không copy logic.
  2. Dùng $NT_WORKSPACE cho mọi path tuyệt đối — Phase 0 resolve find-nt-workspace.sh ngay đầu skill. Không hardcode đường dẫn tuyệt đối.
  3. Command = thin wrapper → Skill — Command chứa 3-5 dòng delegate, không chứa logic thật.
  4. Hook Kiro dùng ngôn ngữ tự nhiên — Kiro không chạy bash trực tiếp. Viết prompt rõ điều kiện + hành động, AI sẽ thực thi.
  5. Test từng IDE sau khi viết — Sau khi tạo SKILL.md, thử invoke từ cả 3 IDE để verify wrapper path đúng.
