🔌 ACP Hermes

Hiểu đúng ACP (Agent Client Protocol) là gì và vì sao Rider + Zed đều dùng chung Hermes/9Router qua nó

WHATLà gì?

ACP (Agent Client Protocol) là chuẩn giao tiếp giữa IDE và một AI agent chạy local. Hermes implement chuẩn này (hermes acp) để Rider và Zed có thể spawn Hermes như backend chat ngay trong IDE, dùng chung 1 config ~/.hermes/config.yaml và 1 route 9Router phía sau.

WHYTại sao dùng?

Không cần cấu hình model/API key riêng cho mỗi IDE. Đổi model hay đổi key 1 lần trong Hermes (qua hermes setup) là cả Rider và Zed đều dùng được route mới — tránh tình trạng "401 ở chỗ này nhưng chỗ khác vẫn chạy" do config lệch nhau.

WHOAi dùng?

Dev dùng cả 2 IDE Rider (cho .NET BE) và Zed (cho task nhẹ/nhanh), muốn chat AI trực tiếp trong panel của IDE thay vì mở terminal Hermes riêng.

WHENKhi nào dùng?

• Mở thread chat mới trong Rider/Zed để hỏi nhanh, không rời IDE
• Sau khi đổi model/provider bằng hermes setup — luôn cần mở thread mới, process ACP cũ không tự reload config
• Khi cần debug 401 — đọc log trước khi đổi config

WHEREỞ đâu?

• Config dùng chung: ~/.hermes/config.yaml
• Log debug: ~/.hermes/logs/agent.log (xem qua hermes logs -f)
• Rider: Settings của plugin Hermes Agent
• Zed: Agent Panel → ACP, trỏ tới lệnh hermes acp

HOWCách dùng

Setup / đổi model

hermes setup

Wizard chọn endpoint + model, dùng chung cho cả Rider & Zed

Test nhanh qua CLI

hermes -z "hello em"

Verify config trước khi mở IDE, đỡ mất công debug trong UI

Xem log khi lỗi

UI Rider/Zed im lặng khi 401 — log mới thấy nguyên nhân thật

🔍 Root cause

2 lớp lỗi 401 chồng lên nhau, dễ nhầm là 1 lỗi duy nhất.

Lớp 1 — Session ACP cũ dùng config đã hỏng

Thread Zed khởi động trước lúc đổi model bằng hermes model. Process ACP không hot-reload config giữa session, nên vẫn kẹt ở route 9Router cũ (key invalid).

Fix: đóng thread cũ, mở thread mới để spawn process mới đọc config hiện tại.

Lớp 2 — Key OpenRouter trực tiếp cũng bị reject

Sau khi đổi sang provider: openrouter trực tiếp (không qua 9Router), vẫn 401 "User not found" — key OpenRouter trong config không hợp lệ với account.

Fix: quay lại dùng route qua 9Router (key 9Router còn sống), không phụ thuộc trực tiếp vào OpenRouter key nữa.

💡 Tại sao dễ nhầm lẫn

Cả 2 lỗi đều trả về HTTP 401, nhìn rất giống nhau trên UI Zed (chat panel trống, không gì hiện ra). Chỉ phân biệt được khi đọc kỹ base_url + message chi tiết trong ~/.hermes/logs/agent.log.

🛠️ How to Fix — Step by Step

Toàn bộ qua hermes setup — không cần sửa file config tay.

# 1. Xem log trước để xác định đúng lỗi
hermes logs -f

# 2. Mở wizard setup
hermes setup
# → chọn "custom (direct API)"
# → API base URL: http://localhost:20128/v1   (gateway 9Router local)
# → API key: nhập key 9Router của bạn (lấy ở dashboard 9Router, KHÔNG share công khai)
# → đợi "Verified endpoint ... (N model(s) visible)"
# → Select API compatibility mode → giữ "Chat Completions" (Enter)
# → Select model → chọn model bạn cần (ví dụ Combo-Company)
# → Context length → để trống, Enter (auto-detect)
# → Display name → để trống, Enter (giữ mặc định)

# 3. Test ngay bằng CLI, không cần qua Zed
hermes -z "hello em"

# 4. Quan trọng: đóng thread Zed cũ, mở THREAD MỚI
# (process ACP cũ không tự reload config, phải spawn mới)

📌 Lessons

Process ACP không hot-reload config

Đổi model/provider xong nhớ mở thread mới trong Zed/Rider, đừng tiếp tục gửi trong thread cũ.

401 không có nghĩa là cùng 1 nguyên nhân

Luôn đọc base_url trong log để biết đang fail ở route nào (9Router local vs. provider trực tiếp).

hermes logs -f là bước đầu tiên

UI Zed im lặng khi lỗi — log chi tiết luôn ở ~/.hermes/logs/agent.log.

Không bao giờ paste key/token thật ra ngoài

Khi debug config có chứa secret, che giá trị bằng *** trước khi gửi đi bất kỳ đâu (chat, log công khai, repo).
